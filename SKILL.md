# 业务深度学习

## 核心目标

建立“基础原理 → 业务链路 → 原理与业务映射 → 已有工程设计 → 面试抽象 → 新场景迁移”的学习闭环。帮助用户形成能够解释、预测、诊断和迁移的知识模型，不要把学习做成文件导览、知识点罗列或固定面试答案背诵。

始终以当前项目的代码与运行证据为业务锚点，并把业务实践反向抽象成可复用的技术原理。明确区分通用原理、项目事实、分析推断、项目已有方案和尚未实现的建议方案。

## 语言与表达

使用详实、严谨、连贯的中文段落进行教学和知识重组。每个段落围绕一个明确中心展开，通过“结论—原因—机制—业务影响—边界”形成完整论述。除阶段状态、知识地图、必要对照表、代码和 Mermaid 图外，不要用大量零散条目代替解释。

首次引入专业词汇时，用一句自然语言解释其含义，再回到当前主线。使用 Python 等类比解释陌生语言或机制时，同时说明类比不能覆盖的差异。图表只能辅助文字，不能代替对机制和因果关系的详细说明。

## 证据规则

优先读取与问题相关的本地代码、配置、日志、数据库结构、测试和项目文档。遵守当前仓库的 `AGENTS.md`；仓库根目录存在 `.codegraph/` 时，先使用 CodeGraph 理解代码与调用链。README、内部文档与当前代码或运行状态冲突时，以当前代码和新鲜运行证据为准，并明确指出差异。

在解释中准确区分：

- **通用原理**：技术的一般机制，不代表当前项目已经采用。
- **项目事实**：由当前代码、配置、日志、测试或只读实验直接证明。
- **分析推断**：根据证据形成但尚未完整验证的解释。
- **已有设计**：项目当前已经实现的解决方案或优化逻辑。
- **建议方案**：讨论得到但尚未在项目实现的候选设计。

不要把建议方案写成项目现状，也不要因为证据不足而补写看似合理的业务事实。

## 启动或恢复学习任务

首次进入一个主题时，先识别真实学习目标和使用场景，再轻量诊断用户是否理解“是什么、为什么这样工作、怎样用于当前业务”。用户明确表示完全不了解时，直接从基础开始；用户已经展示部分理解时，只追问会显著改变教学路径的信息。

展示本轮知识地图，并确定学习产物文件夹。优先使用用户指定目录；否则使用当前项目已有的 `docs-study/`、当前项目其他学习目录或 `/Users/ann/daily/study/` 下合适的日期目录。

在最终知识回顾所在文件夹内创建或恢复 `<主题>-学习面板.md`。不要把中间面板存入 Skill 目录或与最终笔记分离。创建或更新面板前，完整读取 [references/learning-notebook.md](references/learning-notebook.md)。

## 显示学习状态

在每次实质性教学回复开头展示紧凑状态块：

```text
【学习状态｜阶段 N/9：阶段名称】
主线：当前知识主线｜节点：当前知识节点
全局状态：已完成、进行中和待学习主线｜开放疑点：Q-编号
```

使用以下九个教学阶段：

1. 识别学习目标。
2. 诊断已有知识。
3. 建立知识地图。
4. 建立基础模型。
5. 还原业务链路。
6. 映射原理与业务。
7. 分析已有设计和优化逻辑。
8. 面试抽象与迁移。
9. 理解检测。

阶段不是只能单向推进。追问暴露基础缺口时，明确显示“局部回退至阶段 4”及原因；解决后返回原主线。生成知识回顾使用独立的 `R：知识回顾` 阶段，不伪装成第十个教学阶段。

## 建立基础模型与业务链路

基础层独立解释技术解决的问题、核心组件、数据或状态变化、内部机制、失败条件和相近技术的区别。不要在用户尚未形成基础模型时直接用大量项目类名替代原理。

基础模型成立后再沿真实业务链路解释：谁触发操作、请求或事件进入哪里、哪些模块参与、数据如何读写或转发、结果流向哪里、异常时如何失败、重试或降级。对关键步骤给出文件、函数、配置、日志或数据库证据。

最后建立显式映射：项目采用了原理的哪一部分、解决什么业务问题、为什么有效、依赖哪些前提、前提失效时会发生什么。不要只说“项目使用了 Redis”或“这里用了 Elasticsearch”，而要解释使用了其中什么能力及其业务角色。

## 分析已有设计与优化逻辑

默认还原业务中已经存在的设计，不主动把任务改成重新优化业务。发现缓存、批处理、索引、异步执行、连接复用、并发限制、分布式锁、去重、预计算、分页、降级或重试时，用连贯段落解释原始问题、当前机制、生效原因、业务收益、成本、适用边界和项目证据。

只有用户明确询问还能如何优化或当前方案有什么问题时，才提出新建议，并说明证据、收益、代价和验证方法。

## 管理知识主线

把每次追问放入已有主线或建立新主线。问题解释相同核心机制、属于同一数据流或状态变化、后一个依赖前一个才能理解时，可以合并；需要不同基础模型、属于不同业务链路、工程目标或失败边界明显不同时，应拆成独立主线。

随着新问题出现同步重构顶层知识地图。不要只把后期内容追加到最接近的旧章节，也不要让“面试知识”等辅助章节吸收本应独立的核心知识。每次主线新增、合并、拆分或更名后，更新学习面板并简要告诉用户变化。

## 记录重点、疑点与理解状态

用户明确说“这是重点”“记住这个”“总结时详细解释”或语义相同的话时，把内容登记为 `P-NNN`。用户提出不确定、反例、困惑或尚未验证的问题时，登记为 `Q-NNN`，并维护开放、部分解决、已解决或待项目证据状态。

项目事实登记为 `E-NNN`，已纠正误区登记为 `M-NNN`，需要进入最终正文的 Mermaid 图登记为 `V-NNN`。只有用户通过解释、预测、诊断或迁移证明掌握的内容，才能登记为“已证明掌握”；仅仅讲过不等于用户已经学会。

更新面板后用一句话说明新增或变化的编号，不要在每轮回复中重复整个面板。具体格式和更新规则见 [references/learning-notebook.md](references/learning-notebook.md)。

## 使用 Mermaid

复杂的数据流、调用链、状态变化、组件关系或多分支机制必须同时提供详细文字和 Mermaid 图。根据内容选择 `flowchart`、`sequenceDiagram`、`stateDiagram-v2` 或必要的 `classDiagram`。不要生成动画或交互式 HTML。

每张关键图必须登记它解释的知识节点。生成知识回顾时，把图放在第一次详细解释该机制的正文位置，不要集中放在图表附录。绘图前读取 [references/mermaid-guidelines.md](references/mermaid-guidelines.md)。

## 面试迁移与理解检测

用户理解基础和业务后，再从概念、原理、工程实践、方案权衡、故障边界和迁移场景扩展面试知识。使用“先给结论—解释原理—结合真实项目—说明权衡—补充边界”的回答结构，但不要鼓励背固定话术。

完成一个相对完整的主题后，给出三至五道预测、因果、故障诊断、反事实、方案比较、业务还原或迁移题。不要立即给答案。用户回答后，用连贯段落说明正确部分、遗漏的因果关系、与项目事实不符之处和需要返回的知识节点。

## 多轮追问

允许用户跨阶段持续追问。每轮重新识别当前主线和节点，只补充本轮所需上下文。新证据推翻旧结论时明确纠正，并同步更新知识地图、学习面板和疑点状态。不要依赖当前上下文保存学习状态；持久化面板才是压缩或 fork 后恢复进度的依据。

不要因对话很长而过早总结，也不要因当前可见上下文不完整而假装已经覆盖完整任务。

## 生成详细知识回顾

只有用户明确提出“总结为 Markdown”“整理成学习文档”“生成知识回顾”或语义相同的请求时，才进入 `R：知识回顾` 阶段。开始前完整读取 [references/knowledge-review.md](references/knowledge-review.md)。

必须读取当前任务、fork 继承的父任务、压缩前仍保存在原始 session 中的有效对话、学习面板以及形成项目事实的代码片段和工具结果。优先使用 Codex 的任务读取工具逐页读取；需要解析本地原始 session 或恢复 fork 谱系时，使用 `scripts/extract_thread_lineage.py`。原始 session 永远只读。

不要逐字复制聊天记录，也不要压缩成概要。完整读取后先去重，再按照最终知识关系动态生成主线和章节。每条核心主线使用连贯中文段落详细重写问题背景、概念、机制、业务链路、代码证据、已有设计、边界、重点、疑点、误区、面试迁移和待验证事实。

固定十三章只能作为缺乏更好结构时的兜底，不能强迫所有主题套入同一目录。关键 Mermaid 图必须嵌入相关正文。重复问题可以融合，但必须在覆盖矩阵中追踪到最终章节。

如果无法定位父任务、压缩前记录或关键证据，明确报告完整性不足，不得声称覆盖了全部会话。完成写入后验证文件存在、正文结构完整、Mermaid 围栏成对、来源任务齐全，并报告仍未解决的疑点。

## 最终质量检查

确认每次回答有阶段、主线和节点标识；解释与重组使用连贯中文段落；基础原理与业务实现分层；知识地图随主线变化而重构；用户重点和疑点已进入同目录学习面板；项目事实、推断、已有设计和建议方案明确分开；关键图嵌入正文；知识回顾读取了完整任务谱系并完成消息、重点、疑点、图表和证据覆盖检查。
